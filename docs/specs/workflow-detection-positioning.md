## 0. The product in one breath

We watch how you work inside an AI session, notice the multi-step thing you
keep redoing, and offer to freeze it into a workflow that runs on its own and
patches itself when your code or context moves.

**Discovery is the wedge. Predictable + self-healing is the promise.**

---

## 1. Positioning statement

> For the **founder-operator who runs their company through AI agents** and is
> tired of re-running the same multi-step agent work by hand, **Workflow
> Detection** watches your sessions, spots the work you repeat, and turns it
> into a **deterministic workflow that runs on its own and keeps itself
> correct**. Unlike record-and-replay macros or hand-authored scripts, the
> workflow is **discovered for you** and **heals itself** as your context
> changes.

One line for the top of a landing page:

> **Your AI just noticed the work you keep repeating.**

## 1A. Real anchor use case: Brex x Nex referral orchestration

The first real workflow this feature crystallizes is the Brex / Navan referral
outreach motion (pilot Phase 3, in-scope for Nex). This is the ground truth the
design and the demo are built on, not a hypothetical.

**The repeating workflow.** For each lead-scored referral opportunity (produced
upstream by Maestro, out of scope), an operator drafts a short personalized Slack
message to the right account owner (AE / CSM / CSE), delivers it (dedupe,
throttle, from "Larissa"), and tracks the referral lifecycle (Identified -> Sent
to owner -> Acknowledged -> Referred) with follow-ups. It runs many times a week,
identical in shape, varying only in data. Textbook detection target.

**Why this validates the positioning:**
- Brex's own pilot success criteria ARE our promise, verbatim: "low failure
  rates, low manual intervention rates, delivery accuracy, tracking
  completeness." That is "predictable workflows that heal themselves," demanded
  and measured by a paying enterprise. We do not have to argue the value.
- The "context-change trigger" from the original idea has a concrete meaning
  here: a new lead-scored opportunity arriving from Maestro fires the workflow.
- The ROI on the aha card must be in OPERATOR currency: hours of manual drafting
  and routing saved, duplicate sends prevented, referrals tracked end to end.
  NOT tokens. Token-cost framing is developer vocabulary and is wrong for this
  hero.

**What this clarifies about the sensor.** For the anchor, detection runs over the
WUPHF office's own agent activity (the referral outreach the office already
does), not a Claude Code session. The generic "Claude Code first" line from the
original idea is the LATER, broader surface. The anchor is WUPHF-office-native,
which is also why the aha card lives in the office (founder's call, 2026-06-16).

**Runtime surfaces the frozen workflow reuses (do not reinvent):**
- Slack send approval -> `ExternalActionApprovalCard` (deterministic-integrations
  work), delivered via Composio. Keeps the human gate, prevents duplicate sends.
- Referral lifecycle tracking -> `DecisionInbox` + lifecycle pills, or a kanban.
- Message drafting -> existing agent + `StructuredMessageCard` rendering.

The only net-new UI is the detection -> creation flow: the aha card and the
draft-review screen. Everything downstream is assembly of shipped components.

## 2. Hero user

**Who:** A technical founder or operator (the WUPHF/Nex ICP, "Sam"-shaped). Runs
lean. Does real operational work through AI agents inside Claude Code: shipping,
triage, research, recurring ops. Comfortable in a terminal, but thinks in terms
of *running the business*, not *writing code*.

**Their day:** They drive an agent through the same 8-to-20-step sequence
several times a week. Every run costs full tokens and full attention, and comes
back slightly different. They know they *should* script it. They never do.

**Why Claude Code is the sensor, not the audience frame:** This person already
does their repeatable agent work in Claude Code, so it is the sharpest place to
*observe* repetition. But the copy speaks their job ("the work your agents
repeat to run your operation"), not a developer's job ("test, fix, PR").

**What they will not tolerate:** an automation they cannot trust to run
unattended, and a tool that nags. Both are positioning constraints, not just UX
notes.

## 3. The villain (status quo)

The competitor is not another product. It is **the re-prompt**.

- They run the same agent dance every week.
- They pay full token cost every single time.
- They get a slightly different result every single time.
- They never find the 20 minutes to script it.

Secondary status-quo workarounds and why they lose:

| Workaround | Why it loses |
|---|---|
| Re-prompting from scratch | Pays full cost + variance every run; zero compounding |
| Hand-authored slash commands / skills | Real work nobody finds time for; rots silently when context changes |
| Zapier / n8n | No AI, no discovery, breaks on change, lives outside the agent |
| Record-and-replay macros | Static; cannot adapt; snaps the moment anything moves |

Name the villain in the copy. "You were never going to find time to script it.
Now you don't have to."

## 4. Category framing

Two honest frames, used for two audiences:

1. **Warm / landing-page frame:** *Your agent's autopilot, written from your own
   work.* Legible, human, leads with the benefit.
2. **Skeptic / technical frame:** *Process mining for AI agents.* The event log
   is your session timeline; we mine it for repeatable processes. Accurate, and
   it gives a credibility anchor to anyone who asks "what is this really."

Lead with #1 everywhere. Keep #2 in the back pocket for technical buyers and
investors.

## 5. Messaging architecture (the funnel)

The product is one thing; the message is staged. Lead acquisition with the aha,
earn retention with the promise, defend with the moat.

**Stage 1 — Hook (acquisition): discovery.**
> Your AI just noticed the work you keep repeating.

**Stage 2 — Promise (retention): predictable + self-healing.**
> Approve it once and it runs the same way every time, on a schedule, by hand,
> or when your context changes, and it updates itself instead of breaking.

**Stage 3 — Moat (expansion / narrative): compounding.**
> Every session your team runs makes your workflows sharper. Automations rot.
> Yours compound.

### Headline

**Your AI just noticed the work you keep repeating.**

### Subhead

It watches your sessions, spots the multi-step thing you've done three times,
and writes it up: goal, steps, outcome. Approve it and it becomes a workflow you
run on a schedule, by hand, or on a context change, and it keeps itself correct
as things move.

### Three proof points (below the fold)

1. **Spotted, not authored.** You don't decide what to automate. It finds the
   pattern and brings you a draft.
2. **Deterministic, not hopeful.** Once frozen, the steps run the same way every
   time. Same cost, same result.
3. **Self-healing, not brittle.** When your code or context changes, it flags
   the drift and proposes the fix instead of silently breaking.

## 6. The aha card (the single most important surface)

The first demo is the "we spotted a workflow" moment mid-session. Discovery-led
positioning lives or dies on this card. **It must read as a painkiller, not a
parlor trick.** That means the card cannot just say "I found a pattern." It must
quantify the payoff on the spot.

Card copy (target shape):

```
┌─────────────────────────────────────────────────────────┐
│  Spotted a workflow you repeat                           │
│                                                          │
│  "Referral outreach to account owners"                   │
│  You've drafted and sent these 5 steps 14 times          │
│  in the last 7 days.                                     │
│                                                          │
│  Goal:    Drive Navan referrals from flagged accounts    │
│  Outcome: Personalized Slack DM to the owner + a          │
│           tracked referral                               │
│  Steps:   5   (review draft ->)                          │
│                                                          │
│  Freeze it and each run is deterministic:                │
│    • ~3 hrs/week of manual drafting + routing back       │
│    • zero duplicate sends, throttle respected            │
│    • fires when a new scored opportunity lands           │
│                                                          │
│  [ Review draft ]   [ Not a workflow ]   [ Later ]       │
└─────────────────────────────────────────────────────────┘
```

Why this defuses the vitamin risk: the card pairs the *surprise* (discovery)
with a *hard number in operator currency* (runs counted + hours saved + dup
sends prevented) in the same glance. The aha and the ROI arrive together.

Visual hierarchy (the design review's main fix). The eye must land in this
order: (1) "Spotted a workflow you repeat" as the quiet label, (2) the workflow
NAME as the largest type, (3) the saving line as the second-loudest element with
the number emphasized, (4) "Review draft" as the single primary button. Goal /
Outcome / Steps are supporting detail at `--text-secondary`. If the saving line
is not the second thing the eye hits, the card is a vitamin again.

Rules for the card:
- Never fire mid-thought; surface at a natural pause or when the office is idle.
- Always show the repeat count and a concrete operator-currency saving. No
  saving, no card (defer to the silent backlog instead of showing a weak card).
- One primary action ("Review draft"). "Not a workflow" and "Later" are quiet
  secondary/tertiary controls. "Not a workflow" trains detection, one click.

## 6A. Aha card — reviewed design spec (from /plan-design-review, 2026-06-16)

This is the design-complete spec for the only net-new surface. Mockup:
`~/.gstack/projects/nex-crm-wuphf/designs/aha-card-20260616/aha-card-final.png`
(token-accurate, light theme, Brex scenario).

**Surface & component.** The card is a **`DecisionInbox` item** (freeze-or-not is
a decision) rendered as a **variant of `StructuredMessageCard`**. Not a bespoke
component. It appears in the referral motion's decision inbox / channel, not a
separate "Spotted" page.

**Provenance (decided).** The repeat count is an **inline expandable proof
chip**: `14 times · since Jun 9 v`. Expanding lists the actual runs (date,
account, owner the DM went to), collapsing back in place. Caps at ~8 rows, then
"+ N more" opens the draft screen. This is the process-mining credibility that
makes a skeptical operator believe the card in one glance.

**Interaction states (the main gap the review closed).**

```
STATE            | WHAT THE OPERATOR SEES
-----------------|--------------------------------------------------------------
detecting        | No card. Needs >=3 confirmed runs AND a computable saving
                 | before any card shows. Detection is silent until then.
no-saving        | No card. Pattern is filed in a quiet "Spotted (backlog)" list
                 | the operator can browse. Never interrupts. (No saving, no card.)
ready (happy)    | The mocked card: eyebrow, name, proof chip, "If you freeze it"
                 | savings, Goal/Outcome/Steps, Review draft / Not a workflow / Later.
proof expanded   | Inline run list; card grows; collapses back.
in review        | "Review draft" opens the draft screen; card marked "in review".
freezing         | Inline "Freezing..." -> resolves to "Workflow created" with a
                 | link to the workflow and its first (supervised) run.
not a workflow   | Collapses to "Dismissed - won't suggest this for 30 days. Undo"
                 | toast; trains the detector. One click, reversible.
later            | Snoozes; reappears next time the pattern runs, max once/week.
detection error  | No card (fail closed). Logged internally; never a broken card.
runtime send err | Handled on the workflow run surface via ExternalActionApproval
                 | Card's existing error states, NOT on the aha card.
```

**Draft-review screen (second net-new surface, high level).** "Review draft"
opens the workflow draft as an **editable Markdown doc** (reuses the wiki/notebook
editor): goal, outcome, the 5 steps with inputs/outputs, the message template,
routing rule, triggers. The full run evidence lives here too. Actions: **Freeze
workflow** (primary) and **Keep editing**.

**Supervised first run (trust ramp).** The first execution after freezing runs
supervised: each Slack send surfaces as an `ExternalActionApprovalCard` the
operator approves, so they watch it work once before it goes unattended. After
one clean supervised run (or "trust it"), subsequent runs fire automatically per
the trigger.

**Themes & tokens.** Verify in `nex`, `nex-dark`, `noir-gold`. Savings block uses
`--success-100/-500`; primary button uses `--accent-warm` (the darker step) for
>=4.5:1 contrast in all three. No hardcoded hex in the real component.

**Accessibility.** Root `role="group"` with an accessible name ("Spotted
workflow: Referral outreach to account owners, seen 14 times"). Proof chip is a
real `<button aria-expanded>`. Check glyphs `aria-hidden`; benefit text carries
meaning without relying on color. Focus order: proof chip -> Review draft -> Not a
workflow -> Later. Enter triggers Review draft; Esc snoozes. Touch targets >=44px
on coarse pointers. Entrance: `opacity 0->1` + `translateY(6px)->0`, ~140ms,
compositor-only; `prefers-reduced-motion: reduce` snaps (matches `web/DESIGN.md`).

**Responsive.** >=480px: the mocked layout. ~320-360px right rail: savings rows
wrap, detail grid stacks label-over-value, action row wraps (Review draft
full-width primary, Not a workflow below, Later right-aligned). Never shrink touch
targets to fit.

**AI-slop fixes applied.** Removed the decorative left-border accent (calm white
surface, App UI rule); production component uses the real `--font-sans` typeface,
not a system stack; one accent, used only on the primary button.

### NOT in scope (design decisions deferred, with reason)

- **Terminal nudge in Claude Code.** Deferred to v1.1; the office is the
  controllable demo surface (founder's call). Heads-down operators miss the
  signal until they open the office; acceptable for the pilot.
- **Multi-workflow / continuous detection.** v1 detects ONE workflow at a time,
  cross-task within a bounded window (see 6B). NOT single-session: the Brex
  pattern lives across ~14 separate tasks, so single-session detection would
  never see it. The "continuously improves from ongoing sessions" expansion is
  later.
- **Silent auto-repair.** v1 self-healing = drift detect + re-propose. No magic.
- **Reporting/analytics surface (Brex step 7).** Tracking exists via
  DecisionInbox + lifecycle; a dedicated analytics view is out of pilot scope.

### What already exists (reuse, do not reinvent)

- `web/DESIGN.md` + `web/public/themes/*.css` — token system, 3 themes, motion
  budget, reduced-motion patterns.
- `StructuredMessageCard` (`web/src/components/messages/cards/`) — the in-chat
  structured card the aha card is a variant of.
- `DecisionInbox` / `DecisionPacketView` (`web/src/components/lifecycle/`) — the
  approval surface the card lives in and the lifecycle pills tracking reuses.
- `ExternalActionApprovalCard` (deterministic-integrations) + Composio — the
  Slack-send approval + delivery for supervised runs and dedupe/throttle.
- Wiki/notebook editor — the draft-review Markdown surface.

## 6B. Architecture spec (from /plan-eng-review, 2026-06-16)

Backend architecture, grounded in a 3-explorer codebase sweep + a Codex outside
voice. Almost everything is assembly of shipped subsystems; the net-new is the
detection miner, the freeze wiring, and the per-opportunity state ledger.

### Data flow

```
WUPHF office agent activity
  [SUBSTRATE CORRECTED by T0] production agents are HEADLESS (claude/codex) and
  bypass the in-process AgentLoop, so output.log is never written (0 files on
  disk). They DO emit tool-level `HeadlessEvent`s (tool_use + a per-turn
  `manifest` with ToolCalls[]{ToolName,Count}; TaskID/TurnID/Agent) -- but that
  stream is IN-MEMORY ONLY today. Phase 0 (T0) persists it to disk.
        v
  [NET-NEW] Detection miner  (cross-task, bounded window)
        - group tool-call sequences by (agent, normalized ToolName seq)
        - cluster identical / near-identical (1-step / retry tolerance)
        - require >=3 repeats AND a computable saving
        - dedup vs existing skills -> UPDATE existing over create new
        - 1 LLM pass: name cluster -> goal / outcome / steps  [->EVAL]
        v
  Candidate -> DecisionInbox "spotted a workflow" card (StructuredMessageCard)
        | human: Review draft -> Freeze
        v
  [REUSE] teamSkill (Status active) + WorkflowKey/Schedule  (broker_types.go:414)
        | [NET-NEW] freeze wiring: idempotent SetSchedulerJob on approve
        v
  [REUSE] scheduler (watchdogScheduler, 20s poll, scheduler.go:324)
        | schedulerJob{TargetType: task}  (constrained agent run)
        v
  [REUSE] headless agent scoped to the frozen skill
        step: LLM draft -> [REUSE] ExternalActionApprovalCard gate -> Composio send
        v
  [NET-NEW] per-opportunity state ledger + lifecycle
        Identified -> Sent -> Acknowledged -> Referred  (dedupe, throttle)
```

### Decisions (this review)

1. **Freeze primitive = reuse `teamSkill`** (active) that auto-registers a
   `schedulerJob`. No 5th entity. The deterministic step spec lives in the
   workflow definition; the skill is provenance + trigger config.
2. **Execution = scheduled constrained agent run** (`TargetType=task`): the job
   kicks a headless agent scoped to the frozen skill, which drafts (LLM) then
   sends through the existing approval gate.
3. **Detection = cross-task deterministic sequence mining + 1 LLM naming pass.**
   NOT single-session. Bounded scan window / since-cursor (stays O(new tasks)).
4. **Triggers v1 = schedule + manual + a scheduled Maestro import.** Real-time
   event trigger (Composio relay) deferred to v1.1.

### Honesty note: what "deterministic" means here

The draft is an LLM call, so this is NOT bit-identical output. The reliability
Brex actually measures is **state-level**: low variance in routing, dedupe,
throttle, delivery, lifecycle tracking, failure handling. So in copy and in
contracts, lead with **"reliable / predictable / no duplicate sends / complete
tracking"**, not "deterministic." The determinism is structural (same steps,
same scoped skill, human-gated send). Overselling "deterministic" to Brex
invites a trust gap (Codex #3/#7). This also tightens the Section 5 promise.

### Hardening folded in from the outside voice (Codex)

- **Clustering contract** (#1): defined grouping key + similarity threshold;
  guard against under-cluster (identical-only) and over-cluster (generic agent
  behavior). This is the highest-risk part of detection-first.
- **Execution state machine** (#8): a durable per-opportunity **send ledger** +
  lifecycle is first-class, not "reuse DecisionInbox." It is what delivers
  dedupe / throttle / delivery accuracy / tracking completeness.
- **Maestro input path** (#5): deferring the real-time trigger hid a dependency.
  v1 needs a **scheduled import/poll** of new scored opportunities with a cursor
  + dedupe, or the scheduled job has nothing to discover.
- **Freeze lifecycle contract** (#6): idempotent upsert; disable job on skill
  disable/archive; version on definition change; restart-safe; concurrent-poll
  safe.
- **Grant scope** (#4): verify the (agent, platform, `SLACK_SEND_MESSAGE`) grant
  + "always allow" actually covers the workflow's send pattern, or
  low-manual-intervention stalls.

### Spike finding + revised phasing (post-spike, 2026-06-16)

A read-only feasibility spike over every real WUPHF home on this machine
CONFIRMED Codex's blocker. The `output.log` tool-call substrate is unpopulated
(1 `test-agent` echo total; active offices write none), and the populated
history (`broker-state.json`: ~41 msgs / 3 tasks / 43 actions) is too coarse to
cluster: 34 of 43 events are a single generic `kind:"automation"`, and there is
no repeated workflow present. Detection-first cannot deliver the pilot on today's
substrate.

**Founder kept detection-first but added a precursor phase** (vision over
de-risk, informed by the evidence). Revised sequencing:

```
Phase 0    Persist the headless tool-event stream to disk. (output.log is
           in-process-only; prod agents are headless and bypass it -- T0.)  (T0)
Phase 0.5  Run/seed the workflow enough times to build a real corpus of
           repeated runs to detect.                                       (T0.5)
Phase 1    Detection miner over the real corpus.                      (T10-T11)
Phase 2    Freeze + execution + pilot.            (T12-T18) + UI (T1-T9)
```

The pilot moves right by Phase 0 + 0.5; detection cannot demo until the corpus
exists. That is the honest cost of detection-first, now budgeted, not hidden.

## 8. Do / Don't

**Do**
- Lead with the moment of being noticed. That is the shareable line.
- Put a number next to the aha every time (runs counted, cost saved).
- Frame self-healing honestly as "flags drift and proposes a fix."
- Tie everything back to "the work that runs your operation."

**Don't**
- Don't say "open source" or "MIT." We are source-available / fair-code (SUL).
- Don't promise silent magic auto-repair in v1. Overclaiming healing kills trust
  the first time it visibly breaks.
- Don't call it a "CRM" or describe Nex as one.
- Don't pitch continuous / multi-workflow intelligence in v1 copy; v1 detects
  one workflow at a time, cross-task within a window (see 6B).
- Don't speak developer-CI vocabulary; the hero is an operator.

## 9. Premises this positioning bets on

State them so we can shoot them down before building.

1. **Operators feel re-prompt toil acutely enough to act.** If they don't, the
   discovery card is a shrug. *Test: do they click "Review draft" the first
   time they see a real card?*
2. **The aha + a hard number reads as a painkiller.** This is the bet that
   directly answers the vitamin risk you accepted. *Test: does the card's saving
   line change "neat" into "yes, do it"?*
3. **Self-healing is real enough in v1 to be claimed.** v1 healing = drift
   detection + re-proposed update, not silent repair. If we cannot ship even
   that, drop "heals itself" from the headline until we can.
4. **"Update existing over create new" is a feature users feel, not just an
   internal dedup.** The compounding promise is only true if users watch a
   workflow they already have get sharper, rather than drowning in near-dupes.

**Biggest risk, named:** #2. You chose the discovery frame knowing the vitamin
risk. The entire defense is Section 6. If the aha card cannot reliably show a
concrete saving, the reliability-led frame becomes the safer headline and we
should revisit.

## 10. Build implications (why we did messaging first)

The positioning forces these product decisions. This is the payoff of locking
the message before the code.

**Must build for v1 (the message promises these):**
- **The aha card** with repeat-count + concrete saving (tokens/time + variance).
  This is surface #1, not a nice-to-have. No card, no product.
- **Cross-task activity capture** over the office's persisted task logs
  (`~/.wuphf/office/tasks/*/output.log`); detection runs over these (see 6B).
- **Cross-task detection** of a repeated multi-step sequence within a bounded
  window (NOT single-session — the Brex pattern spans ~14 tasks).
- **Markdown draft** (goal / outcome / steps) -> reuse the existing
  notebook/wiki human-review gate. Do not build a new review surface.
- **Freeze-to-deterministic execution.** Reuse the workflow-runtime thinking
  already designed (steps / fields / actions / transitions, save-as-skill). The
  net-new part is *detection*, not the runtime.
- **Triggers v1 = schedule + manual + a scheduled Maestro import** that
  discovers new scored opportunities (cursor + dedupe). Real-time context-change
  trigger (Composio relay) deferred to v1.1 (see 6B). For the demo, the
  scheduled import processing new opportunities is the "fires on its own" story.
- **Runtime is assembly, not net-new.** Slack-send approval reuses
  `ExternalActionApprovalCard` (via Composio), tracking reuses `DecisionInbox` +
  lifecycle pills. The only net-new UI is the aha card and the draft-review
  screen.
- **Self-healing v1 = drift detect + re-propose update.** Semi-automatic. Honest
  with the copy.
- **"Update existing over create new":** matching logic so a re-detected pattern
  proposes an update to an existing workflow, not a duplicate. The compounding
  promise depends on this being visible to the user.
- **`/spot-workflow` slash command** for manual trigger, accepting optional
  free-text context to guide detection.

**Explicitly defer (the message does not promise these in v1):**
- Continuous / multi-workflow / ongoing-session detection (the "improves with
  more context" expansion). v1 is cross-task but single-workflow, batch-detected.
- Silent auto-repair. v1 heals semi-automatically.
- Non-WUPHF-office sensors (Claude Code direct, Slack, etc.). v1 sensor is the
  office's own task logs.
- Real-time event trigger (Composio relay ingestion). v1.1.

**Primitive decision (RESOLVED by eng review, see 6B):** a frozen workflow is a
reused `teamSkill` (active) that auto-registers a `schedulerJob` running as a
constrained agent task. No new primitive. Messaging keeps the word "workflow."
