WUPHF adoption artifact

Use HTML when agents need humans to inspect shape, not just prose.

The linked article's useful idea is not "replace markdown everywhere." For WUPHF, the useful move is to add a first-class review artifact format for dense plans, PR explainers, diagrams, reports, and custom editors while keeping the wiki's markdown rebuild contract intact.

Decision

Adopt self-contained HTML artifacts as the preferred format for large agent-produced review surfaces. Do not make HTML the canonical memory or wiki substrate yet. The first implementation is a repo convention, a starter template, and an agent instruction update.

Why this boundary

  • Markdown stays diffable and rebuildable for facts.
  • HTML makes dense reviews more likely to be read.
  • Self-contained files need no app runtime.
  • Interactive artifacts can export back to text or JSON.

Human I/O Thesis

The Karpathy framing makes the rule sharper: audio, pointing, screenshots, and gestures are becoming natural human input to agents, while visual media is the better output channel back to humans. HTML is the useful current bridge because it is still exact and inspectable, but it can carry layout, diagrams, animation, and interaction.

1. Raw text Compact, diffable, but effortful for dense plans and reviews.
2. Markdown Good current default for structured prose and canonical facts.
3. HTML Better for spatial review, comparisons, visual state, and interaction.

Operating Flow

1. Agent reads context Code, docs, git history, Nex context, screenshots, and command output.
2. Agent renders HTML Self-contained page with diagrams, tables, annotated code, and controls.
3. Human reviews visually Open locally, attach to PR, or share as a static file.
4. Artifact exports text Copy PR body, JSON diff, prompt, decision table, or markdown summary.
5. Durable state lands Markdown wiki, code, tests, PR description, or tracked docs record the result.

Where HTML Belongs

Surface Use HTML? Reason Guardrail
Implementation plans Yes Plans often need timelines, dependency maps, data flow, and risk grouping. Keep a compact markdown summary if the decision is durable.
PR review/explainer Yes Diffs, call graphs, and reviewer focus areas are spatial. Do not replace required PR checklist or dispositions.
Design exploration Yes Variants, states, and tokens need side-by-side inspection. Use repo design tokens and include mobile states.
Custom editor Yes Drag, sliders, toggles, previews, and export buttons close the loop. Always include a copy/export path.
Wiki facts and entity logs No The wiki rebuild contract depends on markdown and JSONL as source material. Link HTML from wiki pages; do not make it authoritative.
Short notes No HTML generation cost and noisy diffs are not worth it. Use markdown unless visual structure changes the review outcome.

Starter Artifact Types

Plan Map

Timeline, dependency graph, file ownership, test matrix, and risks.

Use before substantial implementation or multi-agent dispatch.

PR Explainer

Annotated diff, changed-file tour, reviewer checklist, and copyable PR body.

Use for PRs where GitHub's linear diff hides the concept.

System Explainer

Flow diagram, hot path, code snippets, glossary, and gotchas.

Use when onboarding a human or a verification agent.

Throwaway Editor

Purpose-built interface for sorting, tuning, tagging, or selecting data.

Use when text prompting is a poor input device.
