# @wuphf/installer-stub — agent instructions

This package is the WUPHF v1 installer pipeline's **target binary** + the **release workflow** that signs, notarizes, and publishes it. It exists so we can prove the signing + auto-update story end-to-end *today*, before the real desktop shell lands. When `feat/desktop-shell-skeleton` merges and stabilizes, the `electron-builder.yml` `files:` glob retargets to that package and this stub gets deleted.

For the repo-wide base rules (lint, secrets, branch + PR discipline) see the root `CLAUDE.md` / `AGENTS.md`. The rules below are **additional, non-negotiable** for this package.

---

## What this package is

- **`src/main.js`** — A 50-LOC Electron hello-world. Creates one BrowserWindow showing "WUPHF installer-stub v<version>" + the build channel + the auto-update state. That is the entire app.
- **`electron-builder.yml`** — The packaging config: macOS (dmg + zip, hardened runtime, notarized), Windows (nsis, signed via Azure Trusted Signing or signtool), Linux (AppImage + deb, unsigned).
- **`build/`** — Per-platform build resources: entitlements.plist for Mac, certificate bundles, Windows manifest, etc. **No private keys checked in, ever** (per AGENTS.md rule 5 below).
- **`scripts/`** — Helpers: `detect-signing-secrets.sh` (logs which signing secrets are present in env, decides "production-required vs PR-OK"), `verify-appcast.sh` (validates the generated Sparkle appcast.xml against schema), `verify-latest-yml.sh` (validates electron-updater's latest.yml).
- **`.github/workflows/release-rewrite.yml`** — Matrix CI on `macos-14` / `windows-2022` / `ubuntu-24.04`. PR pushes run unsigned dry-runs (proves the build is hermetic). Tag pushes (`v[0-9]+.*-rewrite`) run the full signed + notarized + published path.

This package is **NOT** the real desktop app. It is plumbing. Reviewers reject any change that adds product features here — file an issue against the future `apps/desktop/` instead.

## Hard rules (every PR is checked against these)

1. **Signing fails loud on tag pushes; warns soft on PR pushes.** `scripts/detect-signing-secrets.sh` reads `WUPHF_RELEASE_MODE` (`pr` | `production`). In `production` mode, missing `APPLE_ID` / `APPLE_TEAM_ID` / `APPLE_APP_SPECIFIC_PASSWORD` / `APPLE_CERT_P12_BASE64` / `APPLE_CERT_PASSWORD` (mac) or missing `AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` / `AZURE_SIGNING_ACCOUNT_NAME` / `AZURE_CERT_PROFILE_NAME` / `AZURE_ENDPOINT` (win) exits 1. In `pr` mode it logs a warning and proceeds unsigned. The `release-rewrite.yml` workflow sets the env per `github.event_name`. **Never ship unsigned to a tag push** — the gate is the only thing protecting the trust story.
2. **Notarization is mandatory on macOS for tag pushes.** `electron-builder.yml` enables `mac.notarize: true` and `mac.hardenedRuntime: true`. After notarization, `xcrun stapler staple` runs against the produced .dmg + .zip. The workflow asserts staple-success before upload.
3. **Windows signing uses Azure Trusted Signing.** Not self-signed certs, not EV USB tokens. The signing step uses the official `Azure/trusted-signing-action@<pinned-sha>` (or AzureSignTool with `signtool.exe`). The cert profile name lives in `AZURE_CERT_PROFILE_NAME` env. **No certificate files committed** — the cert is fetched at sign time from Azure.
4. **Auto-update manifests are reproducible.** Sparkle `appcast.xml` and electron-updater `latest.yml` (Win) / `latest-mac.yml` (Mac) are generated by electron-builder. The workflow runs `bash scripts/verify-appcast.sh` and `bash scripts/verify-latest-yml.sh` against the produced manifests, asserting (a) version matches the git tag, (b) signatures present, (c) URLs match the published artifact paths. A divergence fails the release.
5. **Never commit secrets, certs, or signed binaries.** `.gitignore` covers `dist/`, `build/output/`, `*.p12`, `*.pfx`, `*.key`, `*.cer`, `*.mobileprovision`. Pre-push secretlint runs (root lefthook). If a private key ever lands in a commit, **rotate it immediately, rewrite history, and force-push** — the repo is public.
6. **All third-party GitHub Actions pinned to full SHAs.** Per the root `.github/workflows/ci.yml` precedent — `oven-sh/setup-bun@<sha>`, `actions/checkout@<sha>`, etc. Renovate / dependabot may bump them; review the action source before merging the bump. **No `@v3` or `@main` references anywhere in `release-rewrite.yml`.**
7. **electron-builder + electron + electron-updater versions are pinned and aligned.** electron 33+, electron-builder 25+, electron-updater 6+. They have known cross-version compatibility; bumping one without the other is a release-foot-gun. Bumps require a passing dry-run on all three platforms in PR CI.
8. **No skip-notarize, no skip-sign, no `--skip-codesign` flags.** Forbidden anywhere in scripts or workflow yml. CI greps for these literal strings.
9. **The release workflow's `permissions:` block is least-privilege.** `contents: write` only for the publish step (uploads release assets); every other step is `contents: read`. No `id-token: write` unless OIDC is actually used (today it's not — secrets-based).
10. **Reproducible build identifiers.** `electron-builder.yml`'s `appId: ai.nex.wuphf.installer-stub` and `productName: WUPHF (installer stub)` are intentionally distinct from the future real app (`ai.nex.wuphf` / `WUPHF`) so installing this stub does not collide with or auto-update a future production install. When the real shell takes over, `appId` flips to `ai.nex.wuphf` in the same PR that retargets `files:`.
11. **Local builds are unsigned, by design.** `bun run build:dry-run` produces an unpacked dir for fast iteration. `bun run build:current` produces a real installer for the current platform — **unsigned unless the developer has the env vars set locally** (most won't). The `--publish=never` flag is hardcoded in scripts; production publish only happens via the release workflow.
12. **Linux is unsigned.** AppImage + deb. We do not run `dpkg-sig` or `gpg` on these in v1. Document this in `docs/runbooks/linux-distribution.md` so users know to verify via SHA-256 against the GitHub release.
13. **`README.md` and the runbooks match the workflow.** When a signing flag, env var, or step changes, update both `docs/runbooks/{apple-dev-id-setup,azure-trusted-signing-setup}.md` AND the workflow's inline comments in the same PR.

## Verification commands

From repo root unless otherwise noted.

```bash
# Lint
cd apps/installer-stub && bun run lint

# Detect signing secrets (idempotent; safe to run anywhere)
cd apps/installer-stub && WUPHF_RELEASE_MODE=pr bun run check:secrets
cd apps/installer-stub && WUPHF_RELEASE_MODE=production bun run check:secrets  # exits 1 locally unless secrets are exported

# Local dry-run build for the current platform (unsigned)
cd apps/installer-stub && bun run build:dry-run

# Local real installer for the current platform (signed only if local env vars present)
cd apps/installer-stub && bun run build:current

# Demo: produces a runnable .dmg / .exe / .AppImage
bun run installer:build:local
```

The demo deliverable is a `.dmg` (Mac) / `.exe` (Win) / `.AppImage` (Linux) the user can install, double-click, and see "WUPHF installer-stub v0.0.0" in a window. If the install or first-launch fails, the PR is blocked.

## What this package is NOT for

- **Not for product features.** This is a hello-world. Adding routes, panels, IPC, or business logic here is a bug.
- **Not for the desktop shell's security model.** That lives in `apps/desktop/` (`feat/desktop-shell-skeleton`). This stub is intentionally insecure-defaults (sandbox: true is on, but the renderer is empty) — it's not the trust boundary.
- **Not for Linux signing.** AppImage + deb only. Snap, Flatpak, and signed deb come later or never.
- **Not for ARM Windows.** x64 only on Windows for v1. Universal mac (x64 + arm64) is supported.
